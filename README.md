Creating user stories helps clarify those needs by presenting them in simple, goal-oriented language such as “As a user, I want to…”. This approach allows developers to translate general expectations into specific, actionable requirements during development. For example, in the SNHU Travel project, user stories guided the focus toward functionality that enhanced the user’s booking experience rather than merely completing technical tasks. Airgead Banking App - CS 210 - Project Reflection

This project involved developing an interactive banking application for Airgead Banking that demonstrates the power of compound interest. The program allows users to input their initial investment, monthly deposit, annual interest rate, and the number of years to see how their money grows over time. It generates two reports: one showing investment growth without additional monthly deposits and another showing investment growth with monthly deposits. The goal was to help users understand how compound interest affects long-term investments.

I excelled at implementing clean, user-friendly input validation. My program accepts inputs that include symbols such as $ and %, as well as the word “years,” and safely converts them to numeric values. I focused on producing well-formatted output with clear tables and aligned columns, making the reports easy to read and professional in appearance. Additionally, I incorporated pauses and prompts to ensure smooth user interaction.

I could enhance the program by refactoring it into a fully object-oriented design, with classes for the investment profile, calculator, and display renderer. This would improve modularity and make future updates easier. Another improvement would be to optimize compound interest calculations by using more precise rounding and consistent formatting via a helper function. Finally, implementing input loops to allow users to test multiple scenarios without restarting the program would make the app more interactive and efficient.

The most challenging aspect of the project was correctly implementing the compound interest formulas, particularly in distinguishing between monthly and yearly compounding. I initially struggled to obtain an accurate annual interest total. However, I overcame this by breaking the calculation into monthly steps and verifying my results against sample outputs and online compound interest calculators. I also relied on debugging output, C++ documentation, and examples from Zybooks to confirm my logic.

From this project, I gained experience in C++ input validation, formatted console output using <iomanip>, and working with loops and mathematical formulas. These skills are transferable to future programming tasks that require data handling, validation, or formatted reporting. I also strengthened my understanding of modular programming and debugging, which will be valuable in larger software development projects.

To ensure the code is maintainable and readable, I used descriptive variable names, consistent indentation, and clear inline comments to explain the purpose of each function and key formulas. I logically separated tasks into functions for input, display, and calculations, which makes the code easier to test and adapt. Using standard libraries such as <iomanip>, <string>, and <algorithm> improved portability and security by avoiding unsafe C-style input-handling practices.

ABCU Advising Program - CS 300 - Project Reflection

In these projects, I was solving the problem of designing and implementing an advising assistance program for the Computer Science department at ABC University. The goal was to allow advisors to efficiently load course data from a file, store it using an appropriate data structure, and retrieve or display that information in meaningful ways for students. I approached the problem by first analyzing multiple data structures and their runtime and memory tradeoffs, then selecting the structure that best balanced fast lookups with maintainable code. Understanding data structures was critical because the choice directly affected how efficiently the program could search for courses, sort them, and scale as more data was added. By breaking the problem into smaller steps such as file parsing, object creation, storage, and retrieval, I was able to design a solution that aligned with both functional requirements and performance considerations 

Throughout the projects, I encountered roadblocks related to file input handling, data validation, and output formatting, especially when ensuring that user input errors were handled gracefully and that prerequisite data was consistent. I overcame these challenges by revisiting my pseudocode, adding validation checks, and iteratively testing small changes rather than rewriting large sections of code. This project expanded my approach to software design by reinforcing the importance of planning before coding and by showing how early design decisions affect the final implementation. It also changed how I write programs by placing a stronger emphasis on clear structure, meaningful comments, and modular functions. As a result, my code became more readable, easier to debug, and more adaptable to future changes, which are essential qualities for maintainable software in a professional environment.

DriverPass System Analysis and Design - CS 255 - Project Reflection

The DriverPass project focused on designing a web based system for a client called DriverPass, a company that wanted an online platform to help students prepare for driving exams and schedule driving lessons. The system needed to support multiple users, including students, instructors, and administrators, while providing features such as practice tests, lesson scheduling, secure account management, and online package purchasing. Throughout this project, I believe I did particularly well translating business requirements into clear UML diagrams that worked together as a complete system design. My activity, sequence, and class diagrams consistently reflected the same assumptions and workflows, which helped ensure the design was logical, realistic, and easy to understand for both technical and nontechnical audiences.

If I could revise one part of my work, I would refine the presentation diagrams slightly to make them even more streamlined for a client audience, possibly reducing visual complexity while keeping the same functionality. I interpreted user needs by focusing on convenience, security, and clarity, such as allowing users to schedule lessons online, receive immediate feedback from practice tests, and securely manage their accounts. Considering user needs is critical because a system can be technically correct but still fail if it is confusing or inconvenient for the people using it. When designing software, I now approach the process by starting with user goals, breaking workflows into clear steps, and validating those steps through diagrams before considering implementation details. In the future, I would continue using techniques such as use case analysis, process modeling, and iterative feedback to design systems that are both functional and user centered.

Draw It or Lose It Software Design and Systems Architecture – CS 230 – Project Reflection

The Gaming Room was the client seeking to expand their game, Draw It or Lose It, from an Android-only application into a scalable, web-based system accessible across multiple platforms. The software needed to support multiple concurrent game sessions, enforce unique game and team names, assign unique identifiers, and maintain a single authoritative game instance while supporting distributed access. The overall goal was to design a centralized server architecture capable of handling growth, security, and cross-platform compatibility. In developing this documentation, I believe I did particularly well in clearly evaluating operating platforms and justifying the recommendation of a cloud-hosted Linux server. Working through the design document was helpful because it forced me to think about system architecture, storage, memory management, and distributed communication before focusing on code. If I were to revise one part, I would refine the early evaluation section to make the platform comparisons even more concise and tightly aligned to scalability and cost trade-offs.

When interpreting the user’s needs, I focused on concurrency, scalability, and centralized control as the core functional requirements, then mapped those needs directly to architectural decisions such as client-server design, containerization, database integrity constraints, and secure communication protocols. Considering the user’s needs is critical because architectural decisions determine long-term maintainability, performance, and user experience. I approached designing the software by first identifying functional requirements, then analyzing constraints, and finally aligning them with appropriate operating system and distributed system strategies. In the future, I would continue using structured requirement analysis, layered architecture planning, and early evaluation of scalability and fault tolerance to guide development. Establishing architecture before implementation ensures that the code supports the system’s long-term goals rather than simply solving short-term functionality.

Software Testing, Automation, and Quality Assurance – CS 320 – Project Reflection

I can ensure that my code is functional by writing unit tests that verify both valid and invalid inputs. For Project One, I used JUnit tests to confirm that the contact, task, and appointment classes followed the required rules, such as rejecting null values, limiting field lengths, and preventing invalid dates. I can also improve security by validating input, avoiding unnecessary data exposure, and designing the program so users cannot create or modify objects in ways that violate the requirements.

I interpret user needs by breaking the requirements into specific rules that the program must follow. For example, if a user needs a contact record with a required phone number, I translate that into validation logic that prevents the phone number from being null or the wrong length. I also use testing to confirm that each requirement is actually included in the program. This helps ensure the final software matches what the user asked for instead of only working from a developer’s assumptions.

I approach software design by starting with the requirements and identifying the main objects, data fields, and behaviors the program needs. In Project One, I separated the contact, task, and appointment features into individual classes and service classes so each part had a clear responsibility. This made the code easier to test, update, and maintain. I also try to keep the design simple, organized, and focused on meeting the requirements before adding anything extra.
